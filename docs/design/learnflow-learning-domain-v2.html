Learning Domain Overview SCOPE

v2.0 Changes from v1.0
1. Enrollment → CourseEnrollment — explicit name anticipating future MentorGroupEnrollment, EventEnrollment etc.
2. Course.mode removed — replaced with supports_online / supports_offline flags. Delivery format moved to CourseEnrollment.delivery_format. One course, no duplication.
3. Cross-domain soft UUIDs replaced with real ForeignKey — monolith, one DB, no reason to pay the complexity cost early.
4. CourseCategory added — hierarchical catalog taxonomy (2 levels). Single FK on Course.

The Learning Domain owns the complete course structure — from catalog to lesson content. It defines what students learn. It does not own how far they've gotten (UserProgress), how well they did (Assessment), or who taught them (Mentorship).

Learning Flow SEQUENCE

The learning domain defines the structure. Other domains consume it to power the student journey.

1
CourseEnrollment — Student enrolls in Course → CourseEnrollment row created. Delivery format (online/offline) locked here, not on the Course itself.
Learning Domain
2
Lesson Study — Student views LessonContent items (video, text, pdf, etc.) in order.
UserProgress Domain
3
Lesson Practice + Quiz — In-lesson check: LessonPractice exercises and LessonQuiz knowledge check.
Assessment Domain
4
Homework Submission — Student submits work defined by LessonHomework. Mentor reviews via Submissions domain.
Submissions Domain
5
Module Assessment — End-of-module exam. Pass → unlock next Module. Fail → retake. Owned by Assessment domain.
Assessment Domain
Key principle: The Learning Domain defines the structure and content. All student interaction with that content (progress, attempts, submissions, grades) lives in separate domains that reference Learning Domain IDs.

Course TOP-LEVEL CONTAINER

The root entity of the Learning Domain. A Course groups Modules and defines sequencing rules and publication status. It acts as the catalog entry visible to students and admins. Delivery mode is not stored on Course — it is a property of each CourseEnrollment. The Course only declares which delivery formats it supports.

CREATE UNIQUE INDEX idx_course_slug     ON courses_course (slug);

-- Constraint
ALTER TABLE courses_course
    ADD CONSTRAINT chk_course_status  CHECK (status IN ('draft','published','archived')),
    ADD CONSTRAINT chk_course_weeks   CHECK (estimated_weeks IS NULL OR estimated_weeks > 0),
    -- Must support at least one delivery format
    ADD CONSTRAINT chk_course_delivery CHECK (supports_online = TRUE OR supports_offline = TRUE);
Why remove mode from Course? A course like "Python Backend" is one course — its content (modules, lessons) is identical for online and offline students. Putting mode on Course would force duplicate entries: "Python Backend Online" and "Python Backend Offline". Instead, supports_online / supports_offline flags declare capability, while CourseEnrollment.delivery_format records the student's actual delivery choice. No content duplication.
Why is_sequential on Course? Most courses are sequential — students must finish Module 1 before unlocking Module 2. But some courses (e.g. reference material, elective topics) should allow free navigation. This flag drives the unlock logic in the UserProgress domain without needing a separate table.

Module LESSON GROUPING

A Module groups related Lessons into a logical unit. It is the granularity at which students are assessed (end-of-module assessment). Modules are ordered within a course and can be independently published.

courses_module

Ordered lesson grouping within a course · soft-deletable
FieldTypeConstraintsNotes
idUUIDPK
course_idUUIDFK NOT NULL→ courses_course.id, CASCADE DELETE
titleVARCHAR(255)NOT NULLe.g. "Module 3 — Django REST Framework"
descriptionTEXTNULLABLEWhat this module covers. Shown before unlock.
orderSMALLINTNOT NULL, CHECK > 0Position within course. UNIQUE with course_id. No gaps required — reorder via bulk update.
is_publishedBOOLEANNOT NULL, DEFAULT falseUnpublished modules invisible to students. Allows staged rollout of module-by-module.
estimated_hoursSMALLINTNULLABLE, CHECK > 0Shown on module overview screen. Informational only.
created_atTIMESTAMPTZNOT NULL, DEFAULT now()
updated_atTIMESTAMPTZNOT NULL, DEFAULT now()
deleted_atTIMESTAMPTZNULLABLESoft delete
-- Ordering uniqueness per course
CREATE UNIQUE INDEX uq_module_course_order
    ON courses_module (course_id, "order")
    WHERE deleted_at IS NULL;

-- Fast ordered listing for a course
CREATE INDEX idx_module_course_published
    ON courses_module (course_id, "order")
    WHERE is_published = TRUE AND deleted_at IS NULL;
Order uniqueness strategy: The UNIQUE(course_id, order) index is partial (WHERE deleted_at IS NULL). This allows a deleted module to "leave" its slot without blocking a new module at the same position. Reordering is done in a single transaction with a temporary displacement step to avoid constraint violations.

The atomic learning container. A Lesson is what a student opens and works through. It may contain zero or more Content items, zero or one Homework, zero or more Practice exercises, and zero or one Quiz. All four are optional.

Online vs Offline: No mode field on Lesson. The lesson structure (content, homework, quiz) is identical for both delivery formats. The student's CourseEnrollment.delivery_format determines how the course is delivered. For offline courses, the Session Scheduling in the Mentorship domain maps which lessons are covered in which class session.

CourseEnrollment DOMAIN BRIDGE

CourseEnrollment is the foundational link between the Identity domain (User) and the Learning domain (Course). Named explicitly to distinguish it from future MentorGroupEnrollment, EventEnrollment, and CertificationEnrollment models. Its lifecycle is tied to the course — not the user's auth state. It answers: "which students are taking which course, and in which delivery format?"

delivery_format lives here, not on Course. One course can have students enrolled both online and offline simultaneously. The format is a property of the student's enrollment, not of the course content. This eliminates the need for duplicate courses like "Python Backend Online" and "Python Backend Offline".

    ON courses_courseenrollment (user_id, course_id);

-- Fast lookup: all active courses for a student (dashboard)
CREATE INDEX idx_courseenrollment_user_status
    ON courses_courseenrollment (user_id, status);

-- Fast lookup: all students in a course (mentor/admin view)
CREATE INDEX idx_courseenrollment_course_status
    ON courses_courseenrollment (course_id, status);

-- Constraint
ALTER TABLE courses_courseenrollment
    ADD CONSTRAINT chk_enrollment_status         CHECK (status IN ('active','completed','dropped')),
    ADD CONSTRAINT chk_enrollment_delivery_format CHECK (delivery_format IN ('online','offline'));
Why real FK to User? We are building a Django monolith with a single PostgreSQL database. Both accounts and courses apps share the same DB. Real foreign keys give us free cascade/protect semantics, select_related in the ORM, and DB-level integrity at zero extra cost. When (if) we ever extract to microservices, we remove these FKs in a single migration. Premature soft-UUID references add complexity without any benefit today.

CourseCategory CATALOG TAXONOMY

Categories organise the course catalog into a navigable hierarchy. Without this, the catalog becomes a flat, unfiltered list that grows unusable. Supports two levels: root categories (Backend, Frontend, DevOps) and subcategories (Django, FastAPI under Backend).

Single FK, not a junction table. One course belongs to one primary category. A junction table (CourseCategoryAssignment) would be needed only if one course genuinely belongs to multiple categories. Start with a single FK — it covers 95% of cases. If multi-category tagging becomes a product requirement, add a junction table without touching existing data.

courses_coursecategory

Hierarchical catalog taxonomy — max 2 levels (root + subcategory)

parentForeignKeyNULLABLE FKSelf-referential. → courses_coursecategory.id, SET NULL on delete. NULL = root category. Non-null = subcategory.
descriptionVARCHAR(500)NULLABLEShort description shown on catalog filter panel

orderSMALLINTNOT NULL, DEFAULT 0Display order within the same parent. Lower = shown first.
is_activeBOOLEANNOT NULL, DEFAULT trueInactive categories hidden from catalog. Courses retain their category_id — they are not orphaned.

updated_atTIMESTAMPTZNOT NULL, DEFAULT now()

Category hierarchy example

Backend (root, parent=NULL)
  ├── Django        (parent=Backend)
  ├── FastAPI       (parent=Backend)
  └── Node.js       (parent=Backend)

Frontend (root, parent=NULL)
  ├── React         (parent=Frontend)
  └── Vue           (parent=Frontend)

DevOps (root, parent=NULL)
Mobile (root, parent=NULL)
Max depth = 2. The schema allows arbitrary depth (self-referential FK), but the application should enforce max 2 levels (root + one subcategory level). Deeper hierarchies produce confusing UX without product benefit at this stage. Application-level validation: if parent.parent_id is not None: raise ValidationError.

Design Decisions RATIONALE

Single LessonContent table (no table-per-type)

A unified table with a type discriminator + JSONB metadata handles all content types. Adding a new content type (e.g., interactive) is a config change, not a migration. The alternative — 7 separate tables — would require 7 JOINs to load a lesson's content in order.

UNIQUE(parent_id, order) — not gaps, not floats

Integer ordering with a UNIQUE constraint enforces no duplicates at the DB level. Reordering is done in a transaction using a two-step swap (displace to a temp value, then set target). Simpler than floating-point order values, avoids eventual precision collapse.

LessonHomework 1:1, LessonPractice 1:N

One homework per lesson = cleaner student UX (one deadline, one submission per lesson). Multiple practices per lesson = pedagogically correct (several exercises build a skill). Different cardinality = different models of interaction.

Soft delete on Course/Module/Lesson only

Content, Homework, Practice, Quiz are children of Lesson. Soft-deleting the Lesson is sufficient to hide everything below it. Content items themselves are replaced, not archived — hard delete is safe and avoids orphaned records.

Real DB FK while in monolith

Both accounts and courses apps share one PostgreSQL database. Real ForeignKey to AUTH_USER_MODEL gives cascade semantics, ORM select_related, and DB-level integrity for free. Soft UUID references add application complexity with zero benefit in a single-DB setup. When microservices become a reality, remove FK in one migration.

delivery_format on CourseEnrollment, not Course

A single "Python Backend" course can have students enrolled online and offline simultaneously. Putting mode on Course would force duplicate courses — "Python Backend Online" and "Python Backend Offline" — with identical content that must be maintained in parallel. Instead, Course declares supported formats via boolean flags; CourseEnrollment.delivery_format records each student's choice.

CourseCategory as single FK, not junction

One course has one primary category. A junction table (CourseCategoryAssignment) is premature — it adds a join to every catalog query without a clear product need today. The single FK covers 95% of cases. If multi-tagging is required in the future, add the junction table; existing category_id data migrates cleanly.

is_sequential on Course, not Module

Sequential ordering is a course-wide decision. Having it per-module adds complexity without clear benefit — if some modules are free and some sequential, the unlock logic becomes ambiguous. Start simple: course-wide toggle. Per-module overrides are a future extension if the product needs it.

pass_score and max_attempts live in LessonQuiz

These are content configuration values set by the course author, not runtime state. They belong in the Learning Domain alongside the quiz definition. The Assessment domain reads them when creating attempts and computing pass/fail — it never writes them.

deadline_offset_days, not deadline_date

A relative deadline (N days after the lesson unlocks) is far more maintainable than an absolute date. If a course is re-run or a student is given a late-start exception, no homework dates need updating. The absolute deadline is computed at submission time by the Submissions domain: lesson_unlocked_at + offset_days.

Future Extensions NOT IMPLEMENTED

These are identified extension points for the Learning Domain. None are implemented now. The current schema is designed to accommodate them with minimal breaking changes.

📋 Prerequisites

A ModulePrerequisite junction table (module_id + required_module_id) would let admins define "complete Module 2 before unlocking Module 5." Currently the sequential boolean covers 95% of cases.

📝 Lesson Versioning

A LessonVersion table with snapshots of content and metadata. Allows rollback after an edit and tracks what version each student saw. The current model has no version history.

🌍 Localization

A CourseTranslation / LessonTranslation table pattern. The base table stores the primary language; translations store locale + translated fields. The language field on Course prepares for this.

⏰ Content Release Scheduling

A publish_at TIMESTAMPTZ on Module and Lesson. Celery beat job promotes is_published = false → true at scheduled time. Useful for drip-release courses.

🔀 Learning Paths

A LearningPath table with an ordered sequence of Courses. Allows "complete these 3 courses to earn the Backend Developer track" cross-course sequences.

🏷 Multi-Category Tagging

A CourseCategoryAssignment junction table (course_id + category_id) for courses that span multiple domains (e.g., "Full Stack" in both Frontend and Backend). Current single FK covers 95% of cases; add junction only if the product confirms multi-tag need.

🌿 Branching Lessons

Adaptive learning: after a LessonQuiz, route students to different lessons based on score. Requires a LessonBranch conditional routing table. Complex; deferred until product confirms need.

👥 Co-authoring

A CourseAuthor junction table to allow multiple staff members to edit the same course. Currently only created_by_id is tracked.

✅ Content Approval Workflow

A review_status field on LessonContent with states draft → in_review → approved. Required when content is created by contractors or external instructors who need editorial sign-off before publishing.

Design principle applied: Every future extension above can be added as a new table or a new nullable column — none require changing the primary keys, relationships, or constraints of the current schema. This is the mark of a well-normalized foundation.
