Executive Summary STAFF REVIEW

The v1 architecture is structurally sound — the four-table hierarchy, snapshot strategy, and synchronous cascade are the right foundations. However, there are 20 issues ranging from correctness bugs that will surface on day one to scalability problems that only appear under load. None require a full redesign, but several require non-trivial algorithm changes before implementation begins.

Do not implement v1 as written. F2 (lost counter increments) and F18 (bulk attendance race condition) will cause students to get permanently stuck in in_progress with no recovery path except a manual admin override. F8 (ordering bug) will send students to the wrong next lesson on every sequential course. These three alone block implementation.
What v1 gets right: The four-table structure, snapshot-at-enrollment strategy, the synchronous cascade pattern, completion_source as the online/offline differentiator, and post-commit event dispatch are all correct decisions worth keeping. The review below is about fixing the implementation detail, not replacing the foundation.

Snapshot Counter Scalability

F1 O(n) synchronous fan-out when content is deleted Critical

Finding

The handle_content_deleted handler updates required_content_count and viewed_required_count on every LessonProgress row for every enrolled student in the course. It then conditionally calls _check_lesson_completion for each affected row. All of this runs inside the same synchronous event handler — blocking the request thread of whichever staff member deleted the content item.

Impact

A popular course with 10,000 active enrollments triggers 10,000 LessonProgress reads, up to 10,000 LessonContentView existence checks, up to 10,000 counter writes, and up to 10,000 _check_lesson_completion calls — all in one synchronous transaction. This will hit a 30-second DB timeout at a few hundred enrollments. The same problem exists for LessonPublished (adds a new row per enrolled student) and ModulePublished (updates all module progress rows).

Recommendation

Fan-out operations that touch N rows where N grows with enrollment count must be background tasks, not inline event handlers. The event handler enqueues a task; the task processes rows in batches of 500 with a short delay between batches to avoid DB saturation.

v1 — synchronous, will timeout
def handle_content_deleted(lesson_id, content_id):
    affected = LessonProgress.filter(lesson_id=lesson_id)
    for lp in affected:
        lp.required_content_count -= 1
        if viewed(lp, content_id):
            lp.viewed_required_count -= 1
        lp.save()
        _check_lesson_completion(lp)
v2 — enqueue background task
def handle_content_deleted(lesson_id, content_id, was_required):
    if not was_required:
        return

    tasks.fan_out_content_deletion.delay(
        lesson_id=lesson_id,
        content_id=content_id,
    )


def fan_out_content_deletion(lesson_id, content_id):

F2 Race condition on viewed_required_count — lost increments cause permanent stuck state Critical

Finding

The record_content_view algorithm increments viewed_required_count with a read-modify-write pattern. Two concurrent requests (e.g., a mobile app syncing two offline views simultaneously, or a student with two tabs open) will both read the same current value and both write the same incremented value — losing one increment entirely.

The race — thread B's increment is lost
T=0  A reads viewed_required_count = 1
T=0  B reads viewed_required_count = 1
T=1  A writes 2  ← correct
T=1  B writes 2  ← should be 3, is 2

Result: viewed=2, required=3
        content gate never passes.
        Student is permanently stuck.
v2 — use F() expressions + created flag
view_record, created = LessonContentView.get_or_create(
    enrollment_id=enrollment_id,
    content_id=content_id,
    defaults={...}
)

if created and view_record.is_required:

    LessonProgress.filter(enrollment_id=..., lesson_id=...).update(
        viewed_required_count=F('viewed_required_count') + 1
    )

    lp.refresh_from_db()

Why the UNIQUE constraint alone is not enough

The UNIQUE index on (enrollment_id, content_id) prevents duplicate rows but does not protect the counter. The increment must be atomic at the DB level. F() expressions translate to UPDATE ... SET col = col + 1 — a single atomic SQL statement. All counter increments and decrements throughout the entire domain must use this pattern.

Rule for all counters in this domain

Every increment or decrement of viewed_required_count, completed_lessons_count, and completed_modules_count must use Django's F() expression. The only read of these counters allowed in the completion check must be a refresh_from_db() call immediately after the atomic update — not a stale in-memory value.

F3 Counter drift on already-completed lessons High

Finding

When a staff member adds a new required content item to a lesson where some students have already reached status=completed, the fan-out handler increments required_content_count on those completed rows. Now viewed_required_count (3) < required_content_count (4) on a row with status=completed. The CHECK(completed_at IS NULL OR status = 'completed') constraint prevents reverting status, so the row enters a permanently inconsistent state: complete but counters say otherwise.

Recommendation

Fan-out handlers for counter updates must filter out rows where status = 'completed'. Completed lessons are frozen — they do not participate in counter updates regardless of what the Learning Domain changes. A completed lesson stays completed. This is the correct product behaviour: retroactive content addition does not un-complete a lesson a student has already finished.

-- v2: fan-out targets only non-completed rows
UPDATE progress_lessonprogress
   SET required_content_count = required_content_count + 1
 WHERE lesson_id = $lesson_id
   AND status != 'completed';   -- ← this filter is the entire fix
F4 No handler for LessonContentAdded — new required content invisible to counters High

Finding

The v1 document defines a handler for LessonContentDeleted but not for LessonContentAdded. The Learning Domain's LessonContentService.add_content() does not emit a corresponding event that UserProgress listens to. When a staff member adds a new required content item to a published lesson, all enrolled students' required_content_count snapshots become stale — they remain at the old value. Students could complete a lesson without viewing the new required content.

Recommendation

Learning Domain must emit LessonContentAdded(lesson_id, content_id, is_required). UserProgress must handle it with the same fan-out background task pattern as F1. The same status != 'completed' filter from F3 applies.

F5 CHECK viewed ≤ required + 1 is wrong — masks bugs Low

Finding

The constraint CHECK(viewed_required_count <= required_content_count + 1) was documented as "tolerance for concurrent updates." This is the wrong way to handle concurrency. Tolerating a counter that is one ahead hides genuine bugs and will not catch a counter that's 3 or 5 ahead (which can happen with F4 and the fan-out race). Once F2's F() expressions are applied, concurrent increments are atomic and no tolerance is needed.

Recommendation

Remove the +1 tolerance. The correct constraint is CHECK(viewed_required_count <= required_content_count). With atomic F() increments and the status != 'completed' filter from F3, this constraint will never be violated in normal operation. Any violation indicates a real bug that should be surfaced, not masked.

Progress Percentage Calculation

F6 Module-granularity percentage is misleading — equal weight regardless of lesson count High

Finding

The current formula is completion_percentage = completed_modules_count / total_modules_count. This gives equal weight to every module regardless of how many lessons it contains. A course with modules of sizes [20 lessons, 1 lesson, 1 lesson] would show a student as 67% complete after finishing modules 2 and 3 — when they've only completed 2 of 22 lessons.

v1 — module weight: always 1/N
Module 1: 20 lessons  → weight 33%
Module 2:  1 lesson   → weight 33%
Module 3:  1 lesson   → weight 33%

Student finishes M2 + M3 → 67%
But has done 2 / 22 actual lessons.
v2 — lesson-based: weight by lesson count
total_lessons   = SUM(mp.total_lessons_count)
                = 20 + 1 + 1 = 22

completed_lessons = SUM(mp.completed_lessons_count)

percentage = completed_lessons / total_lessons
           = 2 / 22 = 9%   ← accurate

Schema implication

Add total_lessons_count and completed_lessons_count to CourseProgress. These are maintained the same way as on ModuleProgress — incremented via the cascade. Alternatively, compute on-demand by SUM over ModuleProgress rows. For the monolith with proper indexing, the SUM approach is acceptable and avoids adding two more counters to maintain. Prefer the SUM approach initially — add the denormalized counters only if profiling shows the query is slow at the dashboard scale.

Recommended implementation for get_completion_percentage

Query: SELECT SUM(completed_lessons_count), SUM(total_lessons_count) FROM progress_moduleprogress WHERE enrollment_id = X AND NOT is_stale. Single indexed query per enrollment. Add it to CourseProgressSelector.get_completion_percentage(). Cache the result on the CourseProgress row with a cached_percentage INT field (updated on each lesson completion). Dashboard reads the cached field; the SUM query recalculates it.

F7 assessment_pending shows 0% credit for lesson work inside the module Medium

Finding

A module in assessment_pending has completed_lessons_count == total_lessons_count but ModuleProgress.status != 'completed'. With the v1 percentage formula, the entire module contributes 0 to completed_modules_count. A student who finishes all 10 lessons of an 11-lesson course and is waiting to take the assessment sees themselves at exactly the same percentage as when they started the last module. This is a psychological problem — it demotivates students and makes the progress bar feel broken.

Recommendation

The lesson-based percentage from F6 naturally resolves this — lessons inside an assessment_pending module are counted as completed in completed_lessons_count. The percentage rises as lessons complete, regardless of module assessment status. No additional change needed beyond fixing F6.

Continue Learning Strategy

F8 Cross-module lesson_order comparison missing module_order — wrong lesson returned Critical

Finding

The v1 spec for get_next_lesson returns the LessonProgress with status=unlocked and the lowest lesson_order. But lesson_order is scoped per module — every module starts its own ordering from 1. Comparing raw lesson_order across modules is meaningless.

Module 1 (module_order=1): lessons with lesson_order = 1, 2, 3, 4, 5
Module 2 (module_order=2): lessons with lesson_order = 1, 2, 3

Student just finished Module 1, Lesson 5.
Module 2, Lesson 1 (lesson_order=1) is now unlocked.
Module 1, Lesson 5 is now complete.

v1 query: ORDER BY lesson_order ASC → returns Module 1 Lesson 1 (order=1, status=completed)
          or Module 2 Lesson 1 (also order=1)... undefined tie-break.

Correct query: ORDER BY module_order ASC, lesson_order ASC

Recommendation

The query for get_next_lesson must always include module_order as the primary sort key. This requires module_order to be stored on LessonProgress (it already is, as a snapshot field) and the index must cover (enrollment_id, module_order, lesson_order).

F9 in_progress vs unlocked priority is inverted — mid-lesson student is abandoned High

Finding

The v1 spec says: return the first unlocked lesson, fall back to first in_progress if none. This is backwards. unlocked means the student hasn't started yet. in_progress means the student opened the lesson, watched some content, and left. "Continue" should resume the interrupted lesson, not advance to the next one.

v1 — wrong priority
1. First unlocked lesson  ← sends to NEW lesson
2. Fallback: in_progress  ← abandons the student
                              who is mid-way through
v2 — correct priority
1. Most recent in_progress lesson  ← resume
   (ordered by started_at DESC)
2. First unlocked lesson           ← continue
   (ordered by module_order, lesson_order)
3. None → course complete or pending assessment
F10 assessment_pending returns next_lesson: null — dead-end UX High

Finding

When a module is in assessment_pending, there are no unlocked lessons in the course (the first lesson of the next module is still locked, waiting for the current module to complete). The get_next_lesson query returns nothing. The API currently returns next_lesson: null. The student sees a dead "Continue" button with no explanation — they don't know they need to take an assessment.

Recommendation

Add a next_action field to the response. When get_next_lesson finds no lesson but a module is in assessment_pending, return a structured signal the frontend can act on.

-- v2 response shape for get_next_lesson
{
  "next_lesson": null,
  "next_action": {
    "type": "take_module_assessment",   // or "course_complete", or "lesson_unlocked"
    "module_id": "uuid",
    "module_title": "Python OOP",
    "assessment_url": "/assessments/modules/uuid/"  // from Assessment domain
  }
}

The CourseProgressSelector.get_next_lesson method should be renamed to get_next_action and return a typed discriminated union: { type: "lesson", lesson_id } | { type: "assessment", module_id } | { type: "complete" } | { type: "not_started" }.

F11 Non-sequential "continue" returns first-by-order instead of last-active Medium

Finding

For non-sequential courses, all lessons start as unlocked. The current get_next_lesson implementation ordered by (module_order, lesson_order) would return Lesson 1 of Module 1 every time — the student's last-visited lesson is ignored entirely.

Recommendation

Non-sequential courses should use a different "continue" strategy: return the most recently active lesson, whether it is in_progress or the most recently accessed unlocked lesson. Use LessonProgress.started_at DESC as the sort for non-sequential. The selector must branch on Course.is_sequential (read via Learning Domain selector or cached on CourseProgress).

-- Add is_sequential snapshot to CourseProgress for hot-path use
ALTER TABLE progress_courseprogress
    ADD COLUMN is_sequential BOOLEAN NOT NULL DEFAULT TRUE;

Online vs Offline Completion Logic

F12 Attendance bypasses homework gate — offline students skip required work High

Finding

The v1 design states attendance bypasses both content and homework gates. But homework is pedagogically different from watching a video. Content can be covered in a classroom session — the mentor's attendance mark is a reasonable proxy for "the student engaged with the material." Homework submission is a separate act of work product — attending a session does not produce the homework.

Design question that was skipped

The v1 architecture note says "for offline, attendance IS completion." This was stated as a resolved decision but it conflates two distinct concerns: did the student attend the session? (the attendance mark answers this) vs did the student produce the required deliverable? (only homework submission answers this).

Recommendation

Attendance should bypass the content gate only. The homework gate should apply regardless of delivery format. This is a product decision that must be confirmed, but the architecture should be built to support it. The implementation change is minimal:

v1 — both gates bypassed
def handle_attendance_marked(...):
    lp.status           = 'completed'
    lp.completion_source = 'mentor_attendance'
v2 — content gate bypassed, homework gate still applied
def handle_attendance_marked(...):

    lp.viewed_required_count = lp.required_content_count
    lp.completion_source     = 'mentor_attendance'

    _check_lesson_completion(enrollment_id, lesson_id)

This means an offline student whose lesson has required homework still needs to submit it after attending the session. The content gate is passed by the attendance mark; the homework gate still requires a submission event from the Submissions Domain.

F13 No AttendanceRevoked path — mentor mistakes are permanent Medium

Finding

Lesson completion is designed as a terminal state. But offline attendance can be marked in error — wrong student, wrong session, typo. Since completion_source = 'mentor_attendance' locks the lesson as completed with no reversal path, a mentor mistake permanently advances a student past lessons they haven't actually attended.

Recommendation

Two options. Option A (simple): Add an admin-only POST /progress/enrollments/{id}/lessons/{id}/reset/ endpoint backed by a hard reset in LessonProgressService — this is an escape hatch, not a normal flow. Require dual-staff approval in the application layer. Option B (strict): Keep completion terminal but require the Mentorship Domain to emit a CourseEnrollmentService.complete_enrollment() reversal — complex and probably overkill. Option A is sufficient for v1.

Additionally, the AttendanceMarked event must carry an idempotency_key (session_id + enrollment_id) to prevent duplicate processing if the event is delivered twice.

F14 Delivery format change mid-course — completion logic reads stale context Medium

Finding

An admin can change CourseEnrollment.delivery_format from offline to online mid-course. But LessonProgressService.record_content_view determines which gates to apply by reading enrollment.delivery_format at call time. This is correct — it uses the live value. However, lessons already completed as completion_source = 'mentor_attendance' remain completed under the offline model. Future lessons now use online gate logic. The student is in a mixed state that is actually fine for progress tracking, but the completion percentage and analytics will show a mixed completion_source within one enrollment.

Recommendation

This is acceptable behaviour — document it explicitly. Emit an EnrollmentDeliveryFormatChanged event that UserProgress listens to. When received, update a delivery_format snapshot field on CourseProgress. This snapshot is then available to LessonProgressService without requiring a cross-domain lookup on every content view call.

-- Add delivery_format snapshot to CourseProgress
ALTER TABLE progress_courseprogress
    ADD COLUMN delivery_format VARCHAR(10) NOT NULL DEFAULT 'online';
-- Updated via EnrollmentDeliveryFormatChanged event handler

Content Progress Tracking Model

F15 One API call marks a 2-hour video as "viewed" — no watch threshold Medium

Finding

POST /progress/lessons/{id}/content/{id}/view/ records a view and creates the LessonContentView row. For type=text or type=pdf, a single "opened the page" signal is a reasonable proxy for consumption. For type=video or type=recording, this means a student who opens the video player and immediately closes it has "viewed" a 4-hour lecture and passes the content gate.

Impact

For now, this is a product decision as much as a technical one. The architecture should support enforcement without requiring it. The current model stores last_position_seconds but never uses it for gate evaluation.

Recommendation

Add a minimum_watch_ratio field (DECIMAL 0–1, default 0.0) to LessonContent in the Learning Domain. The UserProgress domain stores total_duration_seconds on LessonContentView (snapshotted from LessonContent at view time) and uses it with last_position_seconds to evaluate whether the watch threshold is met. The content gate check becomes:

-- Content gate check for a single content item (video)
def is_content_item_complete(view_record):
    if content.type not in ('video', 'recording'):
        return view_record is not None

    if content.minimum_watch_ratio == 0:
        return view_record is not None

    if view_record is None:
        return False

    watch_ratio = view_record.last_position_seconds / view_record.total_duration_seconds
    return watch_ratio >= content.minimum_watch_ratio

The viewed_required_count counter is then incremented only when the threshold is met, not on first open. This is a non-breaking change — defaulting minimum_watch_ratio = 0 preserves existing behaviour.

F16 is_required flag change on existing content is not handled Medium

Finding

The Learning Domain allows updating a content item — including its is_required flag. Two cases:

Case A (FALSE → TRUE): A student already viewed this content item before it became required. Their LessonContentView row exists. But viewed_required_count was not incremented when it was first viewed (because it wasn't required then). The student now has an invisible deficit — they've watched the video but the counter doesn't reflect it.

Case B (TRUE → FALSE): A student's viewed_required_count includes this item. The item is now not required. Their counter is inflated, and required_content_count will be decremented by the fan-out handler, leaving viewed > required — technically fine for gate evaluation but semantically wrong.

Recommendation

The Learning Domain should emit a LessonContentRequirementChanged(content_id, lesson_id, old_value, new_value) event. UserProgress handles it by: for FALSE→TRUE, checking if a LessonContentView exists for each enrolled student and, if so, incrementing viewed_required_count. Apply the same background task fan-out pattern from F1.

F17 _unlock_next_module never unlocks the first lesson of the next module Critical

Finding

The v1 cascade is: _check_lesson_completion → _unlock_next_lesson → _check_module_completion → _unlock_next_module → _check_course_completion. The _unlock_next_module function is mentioned but never specified. It presumably sets ModuleProgress.status = 'unlocked'. But it does not unlock the first LessonProgress row in that next module. After module unlock, all lessons in the new module still have status=locked. No lesson is unlocked. The student is stuck — no get_next_lesson call will return anything from the new module.

v1 — next module unlocked, lessons still all locked
ModuleProgress(module=2) → status='unlocked'  ✓
LessonProgress(module=2, lesson=1) → status='locked'  ✗
LessonProgress(module=2, lesson=2) → status='locked'  ✗
LessonProgress(module=2, lesson=3) → status='locked'  ✗
→ get_next_lesson returns nothing. Student is stuck.
v2 — _unlock_next_module also unlocks first lesson
def _unlock_next_module(enrollment_id, completed_module_id):
    next_mp = find_next_module(enrollment_id, completed_module_id)
    if not next_mp:
        return

    next_mp.status     = 'unlocked'
    next_mp.unlocked_at = now()


    first_lesson = LessonProgress.filter(
        enrollment_id=enrollment_id,
        module_id=next_mp.module_id,
        lesson_order=1,
        status='locked'
    ).first()
    if first_lesson:
        first_lesson.status     = 'unlocked'
        first_lesson.unlocked_at = now()
        EMIT LessonUnlocked

This is a logic gap, not an edge case. Without this fix, every sequential course breaks at every module boundary for every student.

Additional Findings

F18 Bulk offline attendance creates race condition on module completion counter Critical

Finding

When a mentor marks attendance for a session that covered 5 lessons, the AttendanceMarked event contains lesson_ids: [L1, L2, L3, L4, L5]. The handler loops and calls handle_attendance_marked for each lesson. Each call ultimately calls _check_module_completion, which reads completed_lessons_count, increments it, and checks if all lessons are done.

If these 5 calls run concurrently (either via Celery parallelism or async task workers), they each read the same starting value of completed_lessons_count before any write lands:

Module has 5 lessons. completed_lessons_count starts at 0.
All 5 completion checks run concurrently — no row locking.

Worker 1: reads 0, writes 1, checks: 1 >= 5? No
Worker 2: reads 0, writes 1, checks: 1 >= 5? No   ← lost update
Worker 3: reads 0, writes 1, checks: 1 >= 5? No   ← lost update
Worker 4: reads 0, writes 1, checks: 1 >= 5? No   ← lost update
Worker 5: reads 0, writes 1, checks: 1 >= 5? No   ← lost update

Final completed_lessons_count = 1.
Module never completes.
Student is permanently stuck unless admin manually overrides.

Recommendation

For the cascade itself: The _check_module_completion function must acquire a row-level lock on the ModuleProgress row using SELECT ... FOR UPDATE before reading and incrementing. This serialises concurrent lesson completions for the same module.

For the bulk attendance event: Process lesson completions for a single enrollment serially, not in parallel. The handler serialises completions per enrollment via a per-enrollment distributed lock or by processing the lesson list in-order within a single task.

-- v2: Row-level locking in _check_module_completion
def _check_module_completion(enrollment_id, module_id):
    with transaction.atomic():

        mp = ModuleProgress.objects.select_for_update().get(
            enrollment_id=enrollment_id,
            module_id=module_id
        )

        ...
SELECT FOR UPDATE must be applied consistently throughout the entire cascade chain: _check_lesson_completion, _check_module_completion, and _check_course_completion all need row-level locks on their respective progress rows. This is non-negotiable for correctness in any concurrent environment.
F19 Re-enroll after drop resets progress to zero — product decision not explicit Low

Finding

When a student drops a course and re-enrolls, a new CourseEnrollment is created (new UUID). ProgressInitialisationService.initialise_progress is called for the new enrollment and creates all-fresh LessonProgress rows (starting from locked/unlocked). The student loses all previously completed progress. This may or may not be the intended behaviour — the v1 document never explicitly decided.

Impact

A student who drops at 80% completion and re-enrolls starts at 0%. For online self-paced learning this feels punitive. For offline group-based learning it may be intentional (they join a new cohort).

Recommendation

Make this an explicit product decision and document it. If "preserve progress on re-enroll" is desired, add a previous_enrollment_id FK (nullable) to CourseEnrollment. On initialisation, check if a completed LessonProgress row exists for the user on a previous enrollment for the same lesson, and if so, start those lessons as completed with the original completion_source. This is opt-in behaviour controlled by Learning Domain's re-enrollment business rules.

F20 Admin override has no audit trail in the schema Low

Finding

The override_lesson_complete API requires a reason string. But LessonProgress has no override_reason or override_by_id fields. The reason is received by the API and discarded. When a compliance audit asks "why was this student's lesson manually completed?", there's no answer in the database.

Recommendation

Add two nullable fields to LessonProgress:

override_by_id     UUID NULLABLE  -- actor who performed the override
override_reason    TEXT NULLABLE  -- reason provided at override time

These fields are only populated when completion_source = 'admin_override'. No migration complexity — both are nullable with no backfill needed.
