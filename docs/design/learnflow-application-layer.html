Selectors — Read Layer SECTION 2

Selectors are the only sanctioned way to read Learning Domain data. They own query optimisation (select_related, prefetch_related, annotations), caching strategy, and visibility rules. Views and other services never query models directly.

Design rule: Selectors return querysets or plain Python objects (dataclasses / dicts). They never mutate data. They accept user as a parameter and apply visibility filters internally — callers never pass is_published=True themselves.

CourseCatalogSelector

Handles all public-facing catalog queries. The hot read path — must be cheap and cacheable.

CourseCatalogSelector

Public catalog · fast, filterable, cacheable
Methods
get_published_courses(filters, user) Returns published, non-deleted courses. Applies category/format filters. Annotates with enrolled_count, module_count. Respects supports_online/supports_offline filter if requested.
get_course_card(slug) Lightweight single-course read for catalog card. Returns title, short_description, thumbnail, category, estimated_weeks, supports_* flags.
get_all_categories() Returns active categories ordered by parent/order. Includes course_count annotation per category.
search_courses(query, user) Full-text search on title + description. Returns published courses only. In Phase 1: icontains. Phase 2: pg_search / trigram index.
Inputs
filters category_slug, delivery_format (online|offline), language, page, page_size
Outputs
Queryset / list Annotated Course queryset. Never exposes deleted_at, draft courses, or unpublished modules to unauthenticated callers.

CourseDetailSelector

Full course detail reads used by enrolled students, staff, and course authors. Visibility rules are stricter than the catalog.

CourseDetailSelector

Full course structure · visibility-aware
Methods
get_course_detail(course_id, user) Returns full course + modules + lessons (metadata only, no content). Staff/author sees unpublished. Student sees published-only. Raises NotFound if deleted.
get_course_for_author(course_id, user) Admin/Staff view. Returns all drafts and unpublished items. Raises PermissionDenied if user is not staff or created_by.
get_modules_for_course(course_id, user) Returns ordered, visible modules for a course. Applies published filter based on role.
get_course_by_slug(slug, user) Slug-based lookup used by public URLs. Raises NotFound for unpublished (unless staff).
Visibility Rules
Unauthenticated Published courses only. Free-preview lessons only.
Student (enrolled) Published content only. No drafts.
Staff / Admin All statuses including draft and unpublished.

QuizSelector

Quiz structure reads used by the Assessment domain when creating attempts. The Learning domain only defines the quiz; Assessment records attempts.

QuizSelector

Quiz definition reads · consumed by Assessment domain
Methods

Services — Write Layer SECTION 3

Services own all state mutations. Each service encapsulates one aggregate's business rules. No view or task should write to models directly — all writes go through a service method. Services emit domain events after successful commits.

Transaction rule: Every service method that mutates data runs inside transaction.atomic(). Domain events are dispatched after the transaction commits — never inside it (to avoid consumers reading partially-committed state).

CourseCategoryService

Catalog taxonomy management
Purpose

Manages the course category hierarchy. Admin-only. Validates hierarchy depth and slug uniqueness.

Methods
create_category(data, actor)Creates root or subcategory. Validates: parent exists if given; parent is not itself a subcategory (max depth=2); slug is globally unique.
update_category(category_id, data, actor)Updates name, description, icon, order, is_active. Slug is immutable after creation.
deactivate_category(category_id, actor)Sets is_active=False. Courses keep their category_id — they are not orphaned.
Business Rules
BR-CAT-01Max 2 hierarchy levels. If parent_id is provided, parent must have parent_id=NULL.
BR-CAT-02Slug is immutable once created. Changing it would break URLs and cached references.
BR-CAT-03Deactivating a category with active children deactivates children too.
Events Produced
CategoryCreatedcategory_id, slug, name

CourseService

Course lifecycle · the most critical service in the domain
Purpose

Manages the full Course lifecycle: create → draft → publish → archive → delete. Enforces all invariants on the Course aggregate.

Methods
create_course(data, actor)Creates draft course. Auto-generates slug from title. Validates: supports_online OR supports_offline must be true.
update_course(course_id, data, actor)Updates metadata. Slug is read-only after publish. Mode flags can be changed only if no active enrollments exist for the affected format.
publish_course(course_id, actor)Transitions draft → published. Validates publish readiness (see BR below). Emits CoursePublished.
unpublish_course(course_id, actor)Transitions published → draft. Blocked if active enrollments exist.
archive_course(course_id, actor)Transitions to archived. No new enrollments possible. Existing students retain access.
delete_course(course_id, actor)Soft-delete. Sets deleted_at. Blocked if any active enrollment exists.
Business Rules
BR-CRS-01Publish readiness: Course must have ≥1 published module, each published module must have ≥1 published lesson.
BR-CRS-02Slug is immutable once status is published. Pre-publish it may be updated.
BR-CRS-03Cannot remove supports_offline=True if active offline enrollments exist.
BR-CRS-04Cannot delete a course with any enrollment (regardless of status). Archive instead.
Events Produced
CourseCreatedcourse_id, title, slug, created_by
CoursePublishedcourse_id, title, slug, supports_online, supports_offline
CourseArchivedcourse_id, title

ModuleService

Module lifecycle within a course
Methods
create_module(course_id, data, actor)Creates module. Appends to end of course order by default. Course must not be deleted.
update_module(module_id, data, actor)Updates title, description, estimated_hours, is_published.
reorder_modules(course_id, ordered_ids, actor)Accepts list of module UUIDs in desired order. Validates: all IDs belong to course; no IDs missing. Runs in single transaction using temporary order displacement.
delete_module(module_id, actor)Soft-delete. Cascades visibility (students can no longer see lessons). Does not delete LessonProgress records in UserProgress domain.
Business Rules
BR-MOD-01Cannot publish a module with zero published lessons.
BR-MOD-02Reorder list must contain exactly all non-deleted module IDs for the course — no partial reorders.
Events Produced
ModuleCreatedmodule_id, course_id, title, order
ModulePublishedmodule_id, course_id

LessonService

Lesson lifecycle within a module
Methods
create_lesson(module_id, data, actor)Creates lesson at end of module order. Module must exist and not be deleted.
update_lesson(lesson_id, data, actor)Updates title, description, estimated_minutes, is_free_preview.
publish_lesson(lesson_id, actor)Sets is_published=True. Lesson must have ≥1 content item OR ≥1 component (homework/practice/quiz).
unpublish_lesson(lesson_id, actor)Sets is_published=False. Checks if parent module will become empty — if so, unpublishes module and warns.
reorder_lessons(module_id, ordered_ids, actor)Same pattern as ModuleService.reorder_modules. All non-deleted lesson IDs required.
delete_lesson(lesson_id, actor)Soft-delete. Blocked if UserProgress domain has completed records for this lesson (checked via service call). Staff can force-delete.
Business Rules
BR-LES-01Cannot publish an empty lesson (no content and no components).
BR-LES-02Soft-deleting a lesson does not delete UserProgress records — those are preserved for analytics.
Events Produced
LessonCreatedlesson_id, module_id, course_id, title
LessonPublishedlesson_id, module_id, course_id
LessonDeletedlesson_id, module_id, course_id

LessonContentService

Content items within a lesson
Methods
add_content(lesson_id, data, actor)Appends content item. Validates type-specific required fields (e.g. url required for video; body required for text). Appends to current max order + 1.
update_content(content_id, data, actor)Updates fields. Type is immutable after creation — content must be deleted and re-added to change type.
reorder_content(lesson_id, ordered_ids, actor)Standard reorder pattern.
delete_content(content_id, actor)Hard delete (content items are replaced, not archived). If UserProgress has viewed this item, those records are preserved but the content_id becomes a dangling reference — acceptable.
Business Rules
BR-CNT-01At least one of url or body must be present. Enforced at service + DB level.
BR-CNT-02Content type is immutable after creation.

LessonHomeworkService

1:1 homework definition per lesson
Methods
set_homework(lesson_id, data, actor)Creates or replaces homework definition. Upsert pattern — if homework exists, update it; if not, create it.
remove_homework(lesson_id, actor)Hard-deletes homework definition. Blocked if Submissions domain has any submission against this homework_id. Returns error with submission count.
Business Rules
BR-HW-01Only one homework per lesson. Enforced at DB (UNIQUE) and service layer.
BR-HW-02Cannot remove homework with existing student submissions. Submissions domain must be consulted.

LessonPracticeService

Ordered practice exercises within a lesson
Methods
add_practice(lesson_id, data, actor)Appends practice item. Validates type (coding/written/interactive) and required fields per type.
update_practice(practice_id, data, actor)Updates instructions, hints, max_score, time_limit_minutes. Type is immutable.
reorder_practice(lesson_id, ordered_ids, actor)Standard reorder pattern.
delete_practice(practice_id, actor)Hard delete. Blocked if Assessment domain has attempts against this practice_id.

LessonQuizService

Quiz definition: settings, questions, options
Methods
create_quiz(lesson_id, data, actor)Creates quiz for lesson. Blocked if quiz already exists (1:1 enforced).
update_quiz_settings(quiz_id, data, actor)Updates pass_score, max_attempts, time_limit, shuffle flags, show_correct_after_attempt.
add_question(quiz_id, data, actor)Appends question. Validates type. For single_choice, at least one correct option required. For short_text, no options needed.
update_question(question_id, data, actor)Updates body, explanation, points. Type is immutable after creation.
reorder_questions(quiz_id, ordered_ids, actor)Standard reorder pattern.
add_option(question_id, data, actor)Adds answer option. Validates: for single_choice, only one option may be marked correct.
delete_quiz(quiz_id, actor)Hard-deletes quiz + questions + options. Blocked if Assessment domain has attempts. Staff can force-delete.
Business Rules
BR-QZ-01pass_score must be 0–100 inclusive.
BR-QZ-02single_choice question must have exactly one correct option.
BR-QZ-03multiple_choice question must have ≥2 options with ≥1 correct.
BR-QZ-04Question type is immutable — delete and recreate if type change needed.

QuizCreatedquiz_id, lesson_id, course_id

CourseEnrollmentService

Enrollment lifecycle — the cross-domain write boundary
Purpose

Manages enrollment creation, status transitions, and delivery format. This service is called by students (self-enroll), admins (bulk enroll), and the future Mentorship domain (offline group enroll).

Methods
enroll_student(user_id, course_id, delivery_format, actor)Creates CourseEnrollment. Validates course is published; validates delivery_format against course.supports_*; checks no duplicate enrollment exists.
drop_enrollment(enrollment_id, actor)Sets status=dropped, stamps dropped_at. Actor must be the student or staff.
complete_enrollment(enrollment_id, actor)Sets status=completed, stamps completed_at. Called by UserProgress domain when all module assessments pass. Actor must be system or staff.
change_delivery_format(enrollment_id, new_format, actor)Changes online↔offline. Restricted to staff/admin. Validates new format is supported by course. Emits EnrollmentDeliveryFormatChanged.
Business Rules
BR-ENR-01Cannot enroll in a draft or archived course.
BR-ENR-02Cannot enroll with delivery_format=offline if course.supports_offline=False.
BR-ENR-03Duplicate enrollment (same user+course) is rejected. Return existing enrollment if status=dropped.
BR-ENR-04Only active enrollment can be completed or dropped. Already-completed enrollments are immutable.
Events Produced
StudentEnrolledenrollment_id, user_id, course_id, delivery_format, enrolled_at
EnrollmentDroppedenrollment_id, user_id, course_id, dropped_at
EnrollmentCompletedenrollment_id, user_id, course_id, completed_at

API Contracts SECTION 4

All endpoints use JSON. Auth via Bearer token (JWT). Pagination uses { count, next, previous, results }. Errors use { error, code, detail }. UUIDs in all ID fields.

URL namespace: All Learning Domain endpoints live under /api/v1/. The prefix learning/ is used in this document for clarity but may be omitted in the final router if there is no namespace collision.

Domain Events SECTION 6

Events are dispatched after successful DB commits. In the monolith, Django signals serve as the event bus. In a future service architecture, these map 1:1 to message queue topics. Each event payload is designed to be self-contained — consumers never need to query back into the Learning domain to process the event.

Naming convention: [Entity][PastTense]. All events carry occurred_at: TIMESTAMPTZ and actor_id: UUID (who triggered the action).

UserProgress Domain Dependencies SECTION 7

Before designing the UserProgress domain, these questions must be answered. They are not optional assumptions — each one changes the data model, the event contracts, or both.

Blocking decisions must be resolved before any UserProgress table is designed. Advisory decisions can be deferred to the first implementation sprint but should be decided before writing service code.

🔴 What is the unit of progress?

Does the student complete a Lesson as a whole, or are individual LessonContent items tracked? If content-level: LessonContentProgress(enrollment_id, content_id, viewed_at). If lesson-level: LessonProgress(enrollment_id, lesson_id, status). This determines the primary key structure of the entire UserProgress domain and what LessonContent.is_required means at runtime.

🔴 What unlocks the next lesson?

Options: (A) Student views all is_required content items → auto-unlock. (B) Student submits homework → unlock after mentor approval. (C) Student passes lesson quiz → auto-unlock. (D) Mentor manually unlocks. This decision defines whether LessonHomework and LessonQuiz are gates or just activities. If (B) or (C): UserProgress needs to listen to Submissions and Assessment events respectively.

🔴 What unlocks the next module?

Is it: (A) All lessons in the module completed? (B) A separate end-of-module assessment passed? (C) Both? Currently the design brief says "module assessment" but that lives in the Assessment domain. UserProgress must know whether to wait for an AssessmentPassed event or compute completion from lesson states alone.

🔴 Does online progress === offline progress?

Online students self-pace through lessons. Offline students attend scheduled sessions. Does a student's LessonProgress get set by (A) the student opening the lesson, or (B) the mentor marking attendance for a session that covered that lesson? If (B), UserProgress depends on an AttendanceMarked event from the Mentorship domain. These are two fundamentally different progress architectures.

🟡 Can a student reset progress?

If a student re-enrolls (was dropped, then re-enrolled), does previous progress carry over or reset? This affects whether LessonProgress is keyed by enrollment_id (reset on re-enroll) or user_id + lesson_id (persists across enrollments). The CourseEnrollmentService handles the re-enrollment scenario (BR-ENR-03) but does not yet define what happens to prior progress.

🟡 Who triggers EnrollmentCompleted?

The Learning Domain's CourseEnrollmentService.complete_enrollment exists, but who calls it? UserProgress (when all modules pass)? Assessment (when final exam passes)? This needs a defined orchestrator. Recommendation: UserProgress domain owns course completion logic and calls CourseEnrollmentService.complete_enrollment as an internal service call. But this must be confirmed before UserProgress is designed.

🔵 Progress granularity for Analytics

Analytics domain will want time-series data: when did the student start each lesson? How long did they spend? Are these timestamps stored in UserProgress, or does Analytics maintain its own event log? Decide whether UserProgress is the system of record for timing data or just for completion state. This affects how many columns are on LessonProgress.

🔵 Handling deleted lessons in active courses

If a staff member soft-deletes a lesson that an enrolled student has already completed, what happens to their progress? Options: (A) Progress records preserved, lesson shown as "removed" in UI. (B) Lesson deleted + progress records deleted, completion percentage recalculated. The LessonDeleted event is already defined — UserProgress must declare its handler strategy before its schema is finalised.

What the Learning Domain already provides to UserProgress:

✓ StudentEnrolled event — UserProgress initialises on this
✓ LessonPublished event — new lesson becomes trackable
✓ LessonDeleted event — UserProgress must handle stale references
✓ EnrollmentDropped event — UserProgress deactivates on this
✓ EnrollmentSelector.is_enrolled() — UserProgress uses this as a gate check
✓ LessonSelector.get_lesson_content() — source of truth for what must be completed
✓ CourseEnrollmentService.complete_enrollment() — UserProgress calls this when done
✓ Course.is_sequential — UserProgress reads this to decide unlock order
