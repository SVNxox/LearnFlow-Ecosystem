UserProgress Domain Overview SCOPE

Decisions incorporated:
1. Progress unit = LessonProgress (not content-item level)
2. Lesson complete when: all required content viewed AND required homework submitted
3. Next lesson unlocks automatically on lesson completion
4. Next module unlocks when: all lessons complete AND module assessment passed
5. Online and offline share the same model — completion_source records the difference
6. UserProgress triggers EnrollmentCompleted after all modules pass

The UserProgress Domain owns the complete state of a student's journey through a course. It is a consumer domain: it owns no content definitions, no enrollments, no assessments. It only tracks state. Every mutation is triggered by an event from another domain or a direct student action.

Owns

CourseProgress, ModuleProgress, LessonProgress, LessonContentView — all progress state rows, counters, gate flags, and timestamps.

Reads (cross-domain, same DB)

Learning Domain tables at initialisation time to snapshot required_content_count, homework_required, and total_lessons_count. All subsequent checks use only UserProgress tables.

Consumes events from

Learning Domain, Submissions Domain, Assessment Domain, Mentorship Domain (future). All writes are triggered by events or direct student API calls (content view).

Produces events for

Notification Domain, Analytics Domain, Certification Domain (via CourseCompleted → Learning's EnrollmentCompleted).

Hot path read pattern: A student opening a lesson triggers: (1) LessonSelector.get_lesson_detail from Learning Domain, (2) LessonProgressSelector.get_lesson_progress from UserProgress. These are two separate domain reads. They must never be merged into a god-query — each domain owns its data.

Database Tables SECTION 2

Four tables. Each mirrors one level of the Course → Module → Lesson hierarchy plus a lightweight content view tracking table.

Snapshot strategy: At enrollment time, UserProgress snapshots structural counts from the Learning Domain (total_lessons_count, required_content_count, homework_required). After that, progress checks read only UserProgress tables — no cross-domain queries in the hot path. Snapshots are updated when Learning Domain emits content change events.

Progress Calculation Strategy SECTION 3

The calculation strategy defines exactly how state transitions happen, what triggers each transition, and how the cascade works from lesson → module → course. Everything is synchronous within a single atomic() transaction. Domain events are dispatched after commit.

State Machines

LessonProgress

Lesson state transitions
locked
──▶prev lesson completed OR first in module (non-sequential: on init)
unlocked
──▶first content view recorded
in_progress
──▶content gate AND homework gate both pass
completed ✓
Admin shortcut: locked | unlocked | in_progress → completed via admin_override (skips gate checks).
Offline shortcut: unlocked → completed via mentor_attendance (both gates bypassed — attendance IS completion).

──▶prev module completed OR first module (on enrollment init)
unlocked
──▶any lesson starts
in_progress

──▶all lessons done + assessment_required=FALSE
completed ✓
in_progress
──▶all lessons done + assessment_required=TRUE
assessment_pending
──▶assessment_passed event received
completed ✓

CourseProgress

Course state transitions
not_started
──▶first lesson view
in_progress
──▶all modules completed → triggers EnrollmentCompleted
completed ✓

Completion Gates

A lesson has two gates (online) or one gate override (offline). Both must pass for lesson completion in online mode.

Content Gate — Online

Passes when required_content_count == 0 (lesson has no required content items) OR viewed_required_count >= required_content_count. The counter avoids cross-domain queries: it's incremented when a required content item is viewed and decremented when a required content item is deleted from the lesson.

Homework Gate — Online

Passes when homework_required == FALSE (lesson has no homework) OR homework_submitted == TRUE. The flag is set by the HomeworkSubmitted event from the Submissions Domain. Gate tracks submission only — grading and review belong to Submissions/Mentor domains.

Attendance Override — Offline

When enrollment.delivery_format == 'offline' and a AttendanceMarked event is received for this student + lesson, both content and homework gates are bypassed. The lesson is immediately marked completed with completion_source = 'mentor_attendance'. The mentor's marking is the authoritative completion signal for offline students.

Assessment Gate — Module Level

Blocks module completion (not lesson completion). Active only when ModuleProgress.assessment_required == TRUE. Passes when ModuleAssessmentPassed event received from Assessment Domain. While in assessment_pending state, the next module remains locked. The assessment itself is owned by Assessment Domain — UserProgress only stores the boolean result.

Completion Algorithms

All three algorithms run synchronously within a single atomic transaction. Events are dispatched post-commit via Django signals.

Algorithm 1 — record_content_view(enrollment_id, content_id)

1. UPSERT LessonContentView(enrollment_id, content_id):
     - IF new row:
         - SET first_viewed_at = now()
         - IF content.is_required:
             - INCREMENT LessonProgress.viewed_required_count
             - IF LessonProgress.status == 'unlocked':
                 SET LessonProgress.status = 'in_progress'
                 SET LessonProgress.started_at = now()
                 UPDATE CourseProgress.started_at (if null)
                 EMIT LessonStarted
     - IF existing row:
         - UPDATE last_viewed_at = now(), view_count += 1
         - UPDATE last_position_seconds (if provided)

2. UPDATE CourseProgress.last_activity_at = now()
   UPDATE ModuleProgress.last_activity_at = now()

3. CALL _check_lesson_completion(enrollment_id, lesson_id)

Algorithm 2 — _check_lesson_completion(enrollment_id, lesson_id) [internal]

lp = LessonProgress.get(enrollment_id, lesson_id)

IF lp.status == 'completed':
    RETURN  -- already done, idempotent guard

content_gate = (lp.required_content_count == 0)
               OR (lp.viewed_required_count >= lp.required_content_count)

homework_gate = (NOT lp.homework_required)
                OR (lp.homework_submitted == TRUE)

IF content_gate AND homework_gate:
    lp.status            = 'completed'
    lp.completed_at      = now()
    lp.completion_source = 'student_activity'

    EMIT LessonCompleted { lesson_id, enrollment_id, course_id, module_id, completed_at }

    -- Cascade step 1: unlock next lesson
    CALL _unlock_next_lesson(enrollment_id, lesson_id)

    -- Cascade step 2: check module
    CALL _check_module_completion(enrollment_id, lp.module_id)

Algorithm 3 — _check_module_completion(enrollment_id, module_id) [internal]

mp = ModuleProgress.get(enrollment_id, module_id)

IF mp.status == 'completed':
    RETURN  -- idempotent guard

-- Increment completed count
UPDATE ModuleProgress
   SET completed_lessons_count = completed_lessons_count + 1
 WHERE enrollment_id = enrollment_id AND module_id = module_id

mp.refresh()  -- re-fetch after update

all_lessons_done = (mp.completed_lessons_count >= mp.total_lessons_count
                    AND mp.total_lessons_count > 0)

IF NOT all_lessons_done:
    RETURN  -- still lessons remaining

-- All lessons done — check assessment gate
IF mp.assessment_required AND NOT mp.assessment_passed:
    SET mp.status = 'assessment_pending'
    EMIT ModuleAssessmentUnlocked { module_id, enrollment_id, course_id }
    RETURN  -- waiting for AssessmentDomain event

-- No assessment required OR already passed
SET mp.status       = 'completed'
SET mp.completed_at = now()
EMIT ModuleCompleted { module_id, enrollment_id, course_id }

CALL _unlock_next_module(enrollment_id, module_id)
CALL _check_course_completion(enrollment_id)

Algorithm 4 — _check_course_completion(enrollment_id) [internal]

cp = CourseProgress.get(enrollment_id)

IF cp.status == 'completed':
    RETURN  -- idempotent guard

UPDATE CourseProgress
   SET completed_modules_count = completed_modules_count + 1
 WHERE enrollment_id = enrollment_id

cp.refresh()

IF cp.completed_modules_count >= cp.total_modules_count AND cp.total_modules_count > 0:
    SET cp.status       = 'completed'
    SET cp.completed_at = now()
    EMIT CourseCompleted { enrollment_id, user_id, course_id, completed_at }

    -- Cross-domain call: Learning Domain
    CourseEnrollmentService.complete_enrollment(enrollment_id, actor='system')

Algorithm 5 — _unlock_next_lesson(enrollment_id, lesson_id) [internal]

current_lp  = LessonProgress.get(enrollment_id, lesson_id)
enrollment  = CourseEnrollment.get(enrollment_id)
course      = Course.get(enrollment.course_id)

IF NOT course.is_sequential:
    RETURN  -- non-sequential: all lessons already unlocked at init

-- Find next lesson in same module (by lesson_order)
next_lp = LessonProgress.objects
           .filter(enrollment_id=enrollment_id,
                   module_id=current_lp.module_id,
                   lesson_order=current_lp.lesson_order + 1,
                   status='locked')
           .first()

IF next_lp:
    SET next_lp.status     = 'unlocked'
    SET next_lp.unlocked_at = now()
    EMIT LessonUnlocked { lesson_id: next_lp.lesson_id, enrollment_id }
    RETURN

-- No more lessons in this module; module will be handled by _check_module_completion

Algorithm 6 — handle_assessment_passed(enrollment_id, module_id) [event handler]

mp = ModuleProgress.get(enrollment_id, module_id)

SET mp.assessment_passed    = TRUE
SET mp.assessment_passed_at = now()

IF mp.status == 'assessment_pending':
    -- All lessons were already done, only assessment was blocking
    SET mp.status       = 'completed'
    SET mp.completed_at = now()
    EMIT ModuleCompleted
    CALL _unlock_next_module(enrollment_id, module_id)
    CALL _check_course_completion(enrollment_id)

-- If status is NOT 'assessment_pending', lessons aren't all done yet.
-- Flag is stored; _check_module_completion will pick it up when lessons finish.

Progress Initialisation

Called by ProgressInitialisationService when StudentEnrolled event is received. Must be idempotent — if called twice, it must not create duplicate rows.

initialise_progress(enrollment_id) [called on StudentEnrolled event]

enrollment = CourseEnrollment.get(enrollment_id)
course     = Course.get(enrollment.course_id)

-- 1. Create CourseProgress (get_or_create, idempotent)
cp = CourseProgress.get_or_create(
       enrollment_id  = enrollment_id,
       course_id      = enrollment.course_id,
       user_id        = enrollment.user_id,
       total_modules_count = Module.count(course_id, is_published=True, deleted_at=None)
     )

-- 2. For each published, non-deleted module (ordered by order):
FOR i, module IN ENUMERATE(modules):
    mp = ModuleProgress.get_or_create(
           enrollment_id     = enrollment_id,
           module_id         = module.id,
           course_id         = course.id,
           module_order      = module.order,
           status            = 'unlocked' IF (i == 0 OR NOT course.is_sequential) ELSE 'locked',
           unlocked_at       = now() IF i == 0 ELSE None,
           total_lessons_count = Lesson.count(module_id, is_published=True, deleted_at=None),
           assessment_required = AssessmentSelector.module_has_assessment(module.id)
         )

    -- 3. For each published, non-deleted lesson in module (ordered by order):
    FOR j, lesson IN ENUMERATE(lessons_in_module):
        is_first = (i == 0 AND j == 0)
        LessonProgress.get_or_create(
          enrollment_id         = enrollment_id,
          lesson_id             = lesson.id,
          module_id             = module.id,
          course_id             = course.id,
          lesson_order          = lesson.order,
          status                = 'unlocked' IF (is_first OR NOT course.is_sequential) ELSE 'locked',
          unlocked_at           = now() IF (is_first OR NOT course.is_sequential) ELSE None,
          required_content_count = LessonContent.count(lesson_id, is_required=True),
          homework_required     = LessonHomework.exists(lesson_id)
        )

NOTE: AssessmentSelector.module_has_assessment() is a cross-domain read at init time only.
All subsequent progress checks use only UserProgress tables.

Selectors — Read Layer SECTION 4

Three selectors. They never mutate state. All return querysets or Python dicts. The most important design rule: selectors must never trigger completion checks. Reading progress must always be side-effect free.

CourseProgressSelector

Dashboard + full course progress tree
Purpose

Provides the student dashboard view and the full per-course progress breakdown. Most frequently called selector — must be heavily optimised.

Methods
get_dashboard(user_id) Returns all active CourseProgress rows for the user, joined with course title + thumbnail, ordered by last_activity_at DESC. Includes completion percentage computed as completed_modules_count / total_modules_count.
get_course_progress(enrollment_id) Returns full tree: CourseProgress + all ModuleProgress rows (with lesson counts) + all LessonProgress rows. Used by the course detail page to render progress state per lesson. Uses prefetch_related.
get_next_lesson(enrollment_id) Returns the single LessonProgress row with status=unlocked and lowest lesson_order across all modules. Drives the "Continue" button. Falls back to first in_progress if no pure unlocked exists.
get_completion_percentage(enrollment_id) Returns an integer 0–100. Uses completed_modules_count / total_modules_count from CourseProgress. If total is 0, returns 0. Single-row read — no joins needed.

LessonProgressSelector

Lesson-level state + unlock status
Purpose

Provides lesson-level progress state used by the lesson view page to show which content items were viewed, whether homework was submitted, and whether the lesson is locked.

Methods
get_lesson_progress(enrollment_id, lesson_id) Returns single LessonProgress row with all gate fields. View layer combines this with LessonSelector.get_lesson_detail() from Learning Domain to build the full lesson page.
get_module_lessons_progress(enrollment_id, module_id) Returns all LessonProgress rows for a module, ordered by lesson_order. Used to render the module sidebar showing which lessons are done, in progress, locked.
get_viewed_content_ids(enrollment_id, lesson_id) Returns set of content_id UUIDs viewed by this student for this lesson. Used by view layer to mark which content items have a "viewed" indicator. Queries LessonContentView.
get_video_position(enrollment_id, content_id) Returns last_position_seconds for a specific content item. Called by video player on load for resume-from-position.
is_lesson_accessible(enrollment_id, lesson_id) Fast boolean check: returns TRUE if status IN ('unlocked','in_progress','completed'). Used as permission gate before serving lesson content. Cached per request.

ProgressAnalyticsSelector

Read models for Analytics domain — never called by student-facing views
Purpose

Provides aggregated progress data to the Analytics domain. These queries are heavier (GROUP BY, annotations) and must never run in the request/response cycle for students. Called by background tasks or Analytics domain services only.

Methods
get_course_completion_stats(course_id) Returns: total_enrolled, total_completed, completion_rate %, average_days_to_complete. Used by Analytics domain to populate course-level metrics.
get_lesson_funnel(course_id) Returns per-lesson: total_started, total_completed, avg_days_to_complete, dropout_rate. Ordered by module + lesson. Used for identifying where students drop off.
get_student_progress_snapshot(enrollment_id) Full progress snapshot for a single student: all LessonProgress + timestamps. Used by mentor dashboards to show a single student's timeline through the course.

Services — Write Layer SECTION 5

Four services, each owning one layer of the hierarchy. All mutating methods run in transaction.atomic(). Domain events dispatched post-commit via transaction.on_commit().

Cascade ownership: Only LessonProgressService may call _check_module_completion. Only ModuleProgressService may call _check_course_completion. No service skips levels. This prevents circular triggers and makes the cascade traceable in logs.

ProgressInitialisationService

Bootstraps all progress rows on enrollment · handles late content additions
Methods
initialise_progress(enrollment_id)Full initialisation from Learning Domain structure. Idempotent (get_or_create). See Algorithm 5 in Section 3. Called by StudentEnrolled event handler.
handle_lesson_published(enrollment_id, lesson_id)Adds a new LessonProgress row (status=locked) to all active enrollments for the course when a new lesson is published. Increments ModuleProgress.total_lessons_count and CourseProgress counts if needed. Called by LessonPublished event.
handle_lesson_deleted(lesson_id)Soft-marks LessonProgress as stale=TRUE (add a is_stale boolean). Does NOT delete progress rows — Analytics needs historical data. Decrements ModuleProgress.total_lessons_count. May trigger _check_module_completion if the deleted lesson was the last incomplete one.
handle_content_deleted(lesson_id, content_id, was_required)If was_required=TRUE: decrements LessonProgress.required_content_count for all affected rows. Also decrements viewed_required_count for rows that had a LessonContentView for this content. Then calls _check_lesson_completion for each affected row — deletion of required content may unblock a student.
handle_assessment_added(module_id)Sets ModuleProgress.assessment_required=TRUE for all active enrollments in this module. If any module is currently in_progress with all lessons done, transitions it to assessment_pending.
Business Rules
BR-INIT-01All initialisation uses get_or_create. Duplicate StudentEnrolled event delivery must not create duplicate progress rows.
BR-INIT-02Stale LessonProgress rows (from deleted lessons) are never deleted. They carry is_stale=TRUE and are excluded from completion counts and UI display but preserved for Analytics.

What Analytics needs that UserProgress does NOT provide

Time-spent per lesson

UserProgress records started_at and completed_at but not active session time. If a student opens a lesson, leaves, and comes back three days later, time-to-complete ≠ time-spent. A session heartbeat mechanism (separate LearningSession table or Analytics-owned) is needed for accurate time-spent. This is deferred — not part of UserProgress v1.

Content type engagement

Analytics may want to know: do students who watch videos complete faster than PDF-only students? This requires combining LessonContentView.content_id with Learning Domain's LessonContent.type. Either Analytics does a cross-domain join, or Learning Domain publishes a content_type snapshot on LessonContentDeleted events.

Per-question quiz analytics

Which quiz questions cause the most failures? This data lives entirely in Assessment Domain. UserProgress only knows "assessment passed/failed" — it has no question-level data. Analytics must subscribe directly to Assessment Domain events (QuizAttempted, QuestionAnswered).

Cohort comparisons

How does offline cohort performance compare to online? Analytics needs delivery_format from CourseEnrollment + completion data from UserProgress. These are in different domains. Analytics should maintain its own denormalized projection combining data from both StudentEnrolled (has delivery_format) and CourseCompleted events.

Analytics access contract: Analytics domain may call ProgressAnalyticsSelector methods directly (same DB, read-only) for bulk historical queries run as background jobs. It must NOT query UserProgress tables directly via raw SQL — all reads go through the selector layer. This ensures query optimisation stays within the UserProgress domain.

Key Design Decisions SECTION 9

Synchronous cascade, not event-driven

Lesson → Module → Course completion happens in a single atomic transaction, not via emitted events. This avoids race conditions, guarantees order, and makes the cascade debuggable in a single stack trace. External domains (Notifications, Analytics) are notified via post-commit events — these are advisory, not causal.

Snapshot counters, not live cross-domain queries

required_content_count and total_lessons_count are snapshotted at enrollment time and maintained via events. This means the completion hot path (checking if a lesson is done) only reads UserProgress tables — zero cross-domain queries in the request/response cycle. The trade-off is event-handler complexity to keep counters in sync.

completion_source = single differentiator for online/offline

The same four tables handle both delivery formats. The completion_source field records whether completion came from student activity, mentor attendance, or admin override. This means all progress queries, analytics, and UI components work identically for both formats — no branching logic in the view layer.

Stale flag instead of delete on lesson removal

When a lesson is deleted from the Learning Domain, its LessonProgress rows are marked is_stale=TRUE rather than deleted. This preserves analytics history, prevents completion percentage from jumping backwards for already-completed students, and allows the deletion to be reverted if the staff member made a mistake. Stale rows are excluded from UI display and completion counts.

Assessment gate at module level, not lesson level

LessonQuiz exists in the Learning Domain but is not a completion gate in UserProgress. The lesson quiz is a learning exercise — it's part of the lesson content experience, handled by the Assessment Domain for attempts/grading. The module-level assessment (a separate entity in Assessment Domain) is the gate that controls module completion. This distinction keeps lesson-level completion simple.

EnrollmentCompleted called post-commit, not inline

CourseEnrollmentService.complete_enrollment() from Learning Domain is called via transaction.on_commit(). This is a cross-domain write call. Running it inside the UserProgress atomic block would create a distributed transaction across two domain service boundaries. Post-commit ensures UserProgress is fully committed before the Learning Domain mutates its own state.
